# Verification-Maximal Capability OS — Revision History

Reverse-chronological revision notes (newest first) for [verification-maximal-os.md](verification-maximal-os.md).

> **Revision note (v17).** This version adds a non-normative companion, **[Userspace Porting Targets](userspace-porting.md)**, enumerating the first userspace applications slated for **source-level re-targeting** and mapping each to the mechanism it lands on (§12, §13, §14, §15) — the userland analogue of the [Generational Directions](generational-directions.md) roster and, like it, carrying no normative weight: every target still enters through the Tier-2/Tier-1 admission discipline of §13, so a listing is intent, not exception. Five targets, all **Rust** (safe Rust is memory-safe by construction, §5/§14, so a `#![forbid(unsafe_code)]` port is the cheapest path to the mandatory Tier-2 memory-safety certificate, §13): **COSMIC Desktop** — shell applets as Tier-2 apps, its `cosmic-comp` compositor promoted to the reference §12 display server (Tier-1, with the §13 information-flow theorems deciding which surface observes which input); **Zed** — a Tier-2 editor whose GPUI renderer collapses onto software V-class compute and whose LSP subprocesses become ring-reached capability compartments (no `fork`/`exec`, §2/§12); **uutils / findutils / diffutils** — adopted as the **seed corpus** for §14's *"reimplemented, not ported"* core utilities (the computational core kept as safe Rust; the POSIX ambient-authority surface — uid/gid, signals, global VFS, `nix`/`libc` `unsafe` — deleted or re-expressed as capabilities, §8), honoring §14 by construction; **Servo** — the §14 per-origin contained browser, whose **JIT-free JS engine is the gating sub-project** (a pure-Rust interpreter, Boa-lineage, is the spec-coherent target since `mozjs` is unverifiable JIT C++ that §14 forbids network-facing; recorded as the defining unresolved tension); and a **new GGUF inference runtime** — not a port (llama.cpp's C++/CUDA base is rejected on §5/§15 grounds) but the concrete build of §12's optional inference server on the M-class GEMM cores (§15), models as content-addressed store objects (§10), sessions zeroized on teardown (§12), its **GGUF container parser a §5 Narcissus verified parser** (attacker-facing model files) and sampling/MoE-routing data-dependence the timing residual to bound or label-fence (§8/§13). The companion also states the shared **porting discipline** — recompile-not-emulate (no shim, no VM, §2/§14) plus the four universal obstacles (`unsafe`→verified-HAL-or-delete §5; GPU→software V/M compute §12/§15; ambient-POSIX→capabilities §8; JIT→pure-interpreter §14) — and the shared prerequisites (the certifying Rust→RV64+CHERI toolchain §18 as the hard no-fallback gate, a software-render library on V-class cores, the WASI-shaped capability libc §14). §14 gains a pointer bullet; no normative section changes and no new §17 residual — the targets *consume* existing mechanisms and residuals, they do not add any. Mirrors v5's introduction of [Generational Directions](generational-directions.md).

> **Revision note (v16).** This version makes the **verified copy-on-write filesystem** normative — promoting §10's mutable-FS sketch (a *"namespace layer over the user-data store,"* Merkle/content-addressed) into a **four-layer verified storage stack** with a named base per layer, the **bcachefs featureset (aside from compression) made verifiable**, and a language bifurcated by the §10 integrity line (§5, §6, §10, §12, §17, §18). Prior versions named only *Perennial/GoJournal-lineage* crash-safety and let bcachefs-class semantics *"fall out"* of content addressing — a content-addressed-store (ZFS-meets-git) framing that misses bcachefs's actual elegance, *"everything is one versioned CoW B-tree."* The stack is now drawn as four Coq-homed layers: **L0** — the **Perennial/GoJournal** journal (Iris/Coq, concurrent crash-safe), the trunk; **L1** — a **unified write-optimized CoW B-tree**, the **VeriBetrFS** B^ε-tree *design re-proved in Coq/Iris* (bcachefs's log-structured 256 KB nodes *are* a B^ε-tree — buffered messages flushed on compaction), one parametric index verified *once* over all key types, a shrink-the-proof dividend of the *"everything is a b-tree"* design; **L2** — FS semantics as typed keys with snapshots a version field *in* the key (O(1) writable snapshots), specified via **AtomFS** (linearizability + crash, Coq) and **(S)FSCQ** (crash refinement); **L3** — **SFSCQ/DiskSec** data-noninterference (Coq). Base selection follows the platform axiom exactly as v10 (seL4) and v12 (crypto) did — *methodology is portable, maturity is not*: **VeriBetrFS's / DaisyNFS's Dafny/Z3 and Yggdrasil's Z3/Rosette are rejected as *bases* on trust-base uniformity** (the Narcissus-over-EverParse, SSProve-over-EasyCrypt ground), their designs/architectures kept and proofs redone in Coq; Yggdrasil is retained as an unverified cross-check, FSCQ as the sequential, Haskell-extracted Crash-Hoare-Logic ancestor superseded by Perennial; **DaisyNFS supplies the top-half-over-GoJournal *layering template*** (its Dafny top-half restated in Coq). **Best-in-class verifiable encryption** is bcachefs's *checksum-is-the-MAC* unification turned into a theorem: **per-extent AEAD** (nonce per extent, Poly1305/GHASH tag *as* the stored checksum) keyed **per confidentiality domain** (not bcachefs's one fs-wide key), keys resident only in the crypto core (§6) so the CT obligation stays on the already-Jasmin core; cipher a §15 choice — **AES-GCM** via `Zvkned`/`Zvkg` (table-free, `Zkt`/`Zvkt`-constant-time) default, **ChaCha20/Poly1305** via `Zvbb`/`Zvbc` for parity — with the *"verifiable"* claim a **composition that exceeds bcachefs**: the §5 three-layer crypto proof (functional ⋈ constant-time ⋈ IND-CPA + INT-CTXT ⇒ IND-CCA reduction, SSProve/FCF) ⋈ the **L3 storage noninterference**, joined at the primitive's functional spec (a new crown-jewel spec, §17). **Compression is dropped — a security gain, not a scope cut:** it deletes the compress-then-encrypt ratio oracle (CRIME/BREACH-class) and removes a decompressor from the read data path (proof-surface shrink), leaving authenticate-then-return. **Language, bifurcated by the integrity line:** the four verified layers are **verified C (CompCert) proved in Coq via Perennial/Iris** with **no managed runtime** — GoJournal's *proof and design* transfer, its Go does not (the GC runtime is banned), Dafny/Z3 and Go both declined; below the line the availability-only block services (replication, EC, tiering, bucket allocator + copygc, FTL) are contained `#![forbid(unsafe_code)]` Rust, a bug or compromise costing availability, never integrity or confidentiality. **Honest residuals (§17):** the ported proofs are **fresh** (a labor-and-freshness cost, the seL4-re-proof class, not a trust cost); **re-homing Perennial/Iris onto CompCert-C** (Goose-for-C / VST, or Gallina→C extraction) is a net-new §18 prerequisite — the *"extract to C"* line made concrete, a delivery risk with no trusted fallback; and the **AE ⋈ noninterference composition seam** at the functional spec is a crown jewel neither side catches. > **Revision note (v15).** This version closes a **constant-time (CT) coverage gap** in the verification story (§5, §6, §13, §15, §17, §18). §5's CT guarantee was a **preservation** theorem — CompCert-CT-class for any primitive on the C path, and Jasmin's Coq-verified-compiler CT-to-assembly for the crypto core — and preservation structurally reaches **only binaries built through a verified compiler**. Everything off that path had **no binary-level CT verifier named**: the FPCC **Islaris "no verified compiler in the loop"** path (§5), and — the larger hole — **every secret-touching Tier-1/2 binary** (§13), because the certifying userspace toolchain (§5) preserves *memory safety*, not constant-time. CT was therefore a property of the binary's **pedigree** (which compiler built it), directly contradicting FPCC's own *"verification is a property of the artifact, not its pedigree"* (§5) — the pedigree-to-artifact move **v9** already made for memory safety, left unmade for timing leakage. The gap is closed by verifying CT **directly on the artifact** against the one §15 `Zkt`/`Zvkt` leakage model, and it **decomposes exactly as RTL ⊑ Sail (v11) and WCET (v13) did — a functional part ⋈ a hyperproperty part**. Constant-time is a **2-safety hyperproperty** the bare *functional* Islaris/Iris-over-Sail logic does not carry, so the **Coq-native closing vehicle** is a **relational (self-composition / 2-safety) program logic over the leakage-annotated Sail semantics** — the binary-level sibling of Jasmin's in-Coq CT proof (ReLoC / relational-Hoare-in-Iris lineage) — emitting an FPCC **constant-time certificate** the §6 checker validates at **zero new prover**. The mature, better-fit tool for the *binary-level* statement is **Binsec/Rel** (relational symbolic execution for constant-time and secret-erasure, directly on the binary against a leakage model), adopted as the **unverified complement / bring-up gate**: like **riscv-formal BMC** (v11) and **aiT** (v13) it is **bounded (path-bounded) symbolic-execution evidence**, untrusted evidence-producing machinery, never the axiom — while **ct-verif** is the LLVM-IR-level sibling, applicable on the compiler path but *not* binary-level for the no-compiler one. The obligation is **scoped, not universal**: CT attaches to any compartment receiving **secret-labeled** material over an IDL confidentiality channel (§12) — the crypto-adjacent and key-handling Tier-1 servers (the radio key hierarchy, §12) and any secret-touching Tier-2 app — never the whole app population, matching the *"tighter guarantees sharpen the holder's stopwatch"* scaling (§17). Edited: this note + §5 (constant-time layer-2 bullet; FPCC-path clause) + §6 (fifth untrusted build artifact) + §13 (Tier-1/2 CT obligation) + §15 (`Zkt`/`Zvkt` = the one leakage model of **both** the preservation *and* the direct-verification paths) + §17 (new constant-time-coverage seam) + §18 (binary-level CT verifier as the fourth crypto-workstream deliverable) + a §19 [Generational Directions](generational-directions.md) subsection. **Honest residual (§17):** Binsec/Rel is bounded evidence, so the relational-Sail-logic certificate is the sole **unbounded** close; CT verification **inherits the RTL ⊑ Sail residual** — the leakage model is sound only once that least-built arrow closes (§15) — and the `Zkt`/`Zvkt` leakage-model statement (already a crown-jewel spec) now carries a **second** consumer; a secret reaching an un-CT-verified compartment is a *labeling*/spec error, not a tooling one.

> **Revision note (v14).** This version elevates the **DRAM (sub-)channel to a first-class spatial-isolation substrate for coherence islands** (§8, §15, §17), sharpening the pre-existing *“DRAM channel and bank partitioning across islands”* (§15) from one knob among several into the **DRAM-tier realization of the island boundary** — completing it coherently from LLC slice → TDM-NoC path → DRAM channel. The enabling fact is **LPDDR6's channel structure** (JESD209-6): the compliant device is **x24 — one 24-bit channel built from two independent 12-bit sub-channels, each with its own narrowed 4-bit CA bus** — narrower and hence *more numerous per module* than LPDDR5X's 16-bit channels and with genuinely independent command streams, so binding disjoint (sub-)channels to disjoint islands **structurally deletes** three cross-island couplings at once — the shared data bus, the shared command/CA bus (the one LPDDR6's per-sub-channel CA newly buys), and row-buffer/bank contention — rather than regulating them. The higher true-(sub-)channel count is the actual lever: **more high-assurance islands can take channel-exclusivity** instead of sharing and falling back to bank-partition + bandwidth-regulation. The binding is adopted as a **graded hierarchy, not a flat one-channel-per-island rule** — strongest to weakest: separate rank/die, then separate 24-bit channel, then separate 12-bit sub-channel, then bank/bank-group within a sub-channel — because **the two sub-channels of a channel sit on the same die and still share refresh, PRAC activation-counting/back-off, thermal, power delivery, and on-die ECC**: sub-channel separation removes the bus/row-buffer channels but *not* die-internal coupling, so high-assurance islands take whole-channel/die exclusivity while mid- and low-sensitivity islands take sub-channel or bank granularity with TME per-domain keys (§15) + bandwidth regulation over the residual. Two costs are booked honestly. **(1)** The channel/sub-channel→island map is enforced by the **memory controller** — a shared resource that never scales (the OPP rule, §15) and sits *below* the islands — so it joins the **TDM-NoC-class non-interference obligations in the Sail model** and lands in the attested static devicetree (§9) as a crown-jewel map (§17). **(2)** Pinning (sub-)channels forfeits interleaving, so each island's bandwidth ceiling **quantizes** to its assigned (sub-)channels and feeds §11 admission — LPDDR6's 12-bit sub-channels make the quantum *finer* (good for island count) but *lower per-sub-channel* (a bandwidth-bound island claims several), and the “100–200 GB/s aggregate” (§15) now distributes as per-island quotas the admission proof allocates. Prerequisite already met: the existing composition-time pinning of LPDDR6's **single-sub-channel dynamic-efficiency mode to static** (§15) is exactly what makes the sub-channel admissible as a boundary — an unpinned reactive sub-channel-occupancy mode would itself be a cross-island channel. §17's **Physical** residual gains the die-internal sub-channel coupling (narrowed by TME + whole-die exclusivity for crown-jewel domains, not eliminated), and the **Specification gap** crown-jewel isolation-model list gains the DRAM channel/sub-channel→island map beside the NoC/coherence model. Edited: this note + §8 timing-side clause + §15 memory-subsystem LPDDR6 and DRAM-pin bullets + §15 LLC/DRAM bullet (the core rewrite) + §15 temporal-isolation and coherence-island touch + §17 Specification-gap and Physical bullets.

> **Revision note (v13).** This version names the **sound WCET-derivation layer** — the analyzer supplying the per-(class, operating-point) execution-time bounds §11 temporal admission consumes — and promotes it from an *unnamed trusted input* into an in-scope, Coq-native mechanization workstream (§5, §6, §11, §15, §17, §18). Prior versions proved *schedulability given WCET numbers* (Prosa-lineage, Coq, §11) but named **no vehicle to derive those numbers soundly**; the whole temporal-admission edifice — OPP/mode selection, TDM NoC schedule, watchdog windows — and much of the §17 timing-channel argument rested on WCET inputs a wrong value silently falsifies, the same *asserted-a-verified-path-where-none-was-named* gap v11 closed for **RTL ⊑ Sail**. The irony was sharp: sound mechanized WCET is barely a field (**aiT/AbsInt** is the commercial standard, unverified; no Coq-native analyzer is in use), yet the design's **in-order + static-only-prediction + fixed-latency** posture exists precisely to make WCET tractable. The layer is now drawn, and it **decomposes exactly as the RTL ⊑ Sail arrow does**. **(a) The low-level micro-architectural model is not a new artifact — it *is* the timing-annotated Sail model of v11 (§15):** the profile's timing discipline (in-order, static-only prediction, fixed-latency DIV/FPU/AMO, Ztso, cache/DRAM/NoC partitioning, TDM NoC, WCET-exact scratchpads, deterministic layout §10) collapses WCET's hard half — aiT's abstract interpretation over pipeline and cache state — into a **per-(class, OPP) latency table** (the fixed-latency mandates *are* its entries) plus cache/fetch/memory terms made a reproducible function of the signed image, sound to the metal by RTL ⊑ Sail (Kami/Kôika); the non-speculative posture was, unstated, a WCET-soundness argument. **(b) The high-level path analysis** — loop bounds + **IPET** (Implicit Path Enumeration Technique) maximizing Σ(block-frequency × block-latency) over the CFG — is a **Coq-verified WCET estimator** (Blazy/Pichardie/Maroneze/Puaut lineage, on CompCert), riding the one Coq prover with Prosa at **zero new trust base**; its ILP solution is a **certificate** the §6 checker validates, so the LP solver — like riscv-formal BMC and the CompCert/Jasmin compilers — is untrusted evidence-producing machinery. The estimator emits a per-task **WCET certificate** composing with the §11 Prosa proof, turning WCET tables from *asserted* into *proven*. **Existence proof:** that verified loop-bound + IPET tool on CompCert already exists — refuting "no Coq-native WCET analyzer" — so the work is retarget-to-RV64GV+CHERI and connect-to-the-timing-annotated-model, not a from-scratch program (the feasibility anchor here, as formosa-crypto is for crypto and srsRAN for radio). Costs booked honestly (§17, §18): the model's **latency magnitudes** become crown-jewel specs (v11 booked only its ordering/2-safety hyperproperties); WCET *inherits* the RTL ⊑ Sail residual, so no sound bound predates that least-built arrow closing; **measurement-based/probabilistic WCET (MBPTA/EVT) is rejected as the admitted bound** (a statistic, not a theorem — the MTE-93% disposition); and **aiT stays the unverified complement**, never the axiom. Unification banked: the same §15 temporal-isolation semantics (fence.t, cache/DRAM/NoC partitioning, static OPP) that delete the cross-domain timing channels are what make single-partition WCET *compose* into the multi-core schedule without an interference term — one non-interference proof both closes the channels and makes the bound sound.

> **Revision note (v12).** This version closes a gap in the cryptography-assurance argument and makes the previously hand-waved "Coq-native ports" destination concrete (§5, §6, §17, §18). The prior crypto posture proved **functional correctness and constant-time** (Fiat-Crypto for classical field arithmetic; libcrux/HACL\* for PQ primitives) but **not cryptographic security** — no game-based **IND-CCA** (KEM) or **EUF-CMA** (signature) reduction to the underlying hardness assumptions — and it *accepted* the F\*/Z3 widening rather than *minimizing* it, the opposite of the trust-base-uniformity move §5 makes picking Narcissus over EverParse for parsers. Crypto verification is now stated as **three composed layers**: (1) functional correctness (implementation refines the abstract scheme), (2) constant-time (data-independent leakage against the §15 `Zkt`/`Zvkt` model, preserved to the metal), and (3) **reduction-level security** (the scheme is IND-CCA/EUF-CMA under named hardness assumptions). Layer 3 was absent and is added **Coq-native via SSProve/FCF** — zero new prover, riding the one Coq kernel (§6) — with **EasyCrypt** as the mature complement (its Why3/SMT backend treated exactly like the F\*/Z3 widening: pragmatic, not Coq-native). For layers 1–2 the constant-time crypto leaves move to **Jasmin** — whose compiler is itself **verified in Coq** and which emits **provably constant-time assembly** — replacing "trust a C compiler to preserve constant-time" with a Coq-verified compiler proof and shrinking the F\*/Z3 implementation surface; **libcrux/HACL\* stay only as the explicit interim** until the Jasmin/Coq-native path lands. The old "until Coq-native ports land" clause now has a named destination: **Jasmin (implementation) + SSProve/FCF (security), EasyCrypt as mature complement.** Costs recorded honestly: a **CHERI-RISC-V Jasmin backend** joins CHERI-CompCert as an in-scope §18 build-artifact workstream (Jasmin targets x86-64/ARM today); the crypto core becomes the one TCB component not written in verified C, though it stays under the **single Coq prover** (§6); the **hardness assumptions** (MLWE/MSIS for PQ; ECDLP/CDH for classical) remain named axioms a reduction *isolates* but cannot *eliminate* (§17); the **security-game specification** joins the crown-jewel spec list — a reduction against a mis-stated game verifies perfectly and proves nothing (§17); and protocol-level composition (TLS/AKA, §12) is acknowledged as a further layer above scheme-level security, not claimed. The **formosa-crypto Kyber/ML-KEM** result (Jasmin implementation + EasyCrypt IND-CCA proof, composed) is the existence proof anchoring feasibility (§18).

> **Revision note (v11).** This version names the **RTL ⊑ Sail refinement layer** — the bottom arrow of G3 (§1: *ISA → modeled hardware → RTL*) — and promotes it from an unstated *trust assumption* into an explicit, in-scope **mechanization workstream** (§1, §15, §17, §18). Prior versions asserted per-feature "RTL-against-Sail proof obligations" (`Zkt`/`Zvkt` data-independent timing, the Ztso store buffer, static-only fetch) as bring-up gates while naming **no vehicle to discharge them**, and §17 listed "Sail fidelity to real silicon" as a bare residual — the one place the end-to-end chain was asserted as if a verified path existed when none was specified. The layer is now drawn. *Functional* refinement (the RTL implements the Sail ISA) is discharged primarily by **Kami/Kôika** (Coq, Bluespec-lineage hardware refinement, MIT PLV), the vehicle that keeps the **single-prover TCB** (§6) — the RTL proof landing in the same Coq as the kernel — with **riscv-formal/rvfi** (YosysHQ, SVA + bounded model checking — what open cores such as Ibex actually ship) as the pragmatic bring-up gate and **Isla** (symbolic execution over Sail) generating the obligations and relating the model to Ztso concurrency. The *timing/ordering* obligations are **hyperproperties** a functional refinement does not by itself carry (constant-time is 2-safety; Ztso is ordering; static-prediction fetch is architectural-state-only), stated against a **timing-annotated** model. The honest ceiling, now stated where it was silent: **riscv-formal BMC is bounded-depth evidence, not an unbounded proof** — so the Kami/Kôika Coq path is the one that *closes* the arrow in-prover, at the price of expressing the cores in a formal-semantics HDL rather than the existing SystemVerilog/Chisel — and **no RTL ⊑ Sail artifact exists for a full application-class core, let alone the heterogeneous §15 die**: this is the **least-built layer of the whole stack** (§18). §17's residual is accordingly **split**: **RTL ⊑ Sail** becomes a *tooling commitment with named residuals* (BMC boundedness, timing-hyperproperty modeling, unbuilt-at-scale), while **fabricated silicon vs. verified RTL** stays the irreducible fab residual (one mask set — §17 silicon supply chain). No trust-base growth: the Coq refinement rides the existing prover; BMC and Isla are untrusted evidence-producing machinery like the compilers (§6).

> **Revision note (v10).** This version reclassifies the kernel *verification route* (§5, §7). The spec already specifies **seL4's design** in every particular — untyped memory, synchronous endpoints + notifications, MCS time-as-capability (§7), derivation-tree revocation and non-interference (§8) — so the former "CertiKOS-lineage kernel proof" named only the **Coq prover**, never a different kernel. Promoting seL4's 2024 program (a verified **multikernel** on multicore; **CHERI-seL4**) to in-scope and spending the platform axiom that engineering is free while trust is scarce, the route becomes an **end-to-end Coq re-verification of the seL4 design, compiled through CompCert/SECOMP**. This keeps the single-prover TCB (Coq — with CompCert, Fiat-Crypto, Perennial, the §6 checker) *and* takes the more mature, more completely proved, longer-scrutinized design; **CertiKOS is demoted from kernel to proof-method lineage** (deep specifications, certified abstraction layers, CompCertX-style verified compilation), its one distinctive asset — proven *shared-memory* concurrency — being dead weight under the share-nothing multikernel (§7), which forbids the shared kernel state that asset verifies. *Adopting* seL4's existing **Isabelle** proof stays rejected (a two-prover TCB). The honest residual: a Coq re-proof is **fresh** — as unbattle-tested as any — so what transfers from seL4's maturity is the design, ABI, C implementation, and long-scrutinized specification, not the proof; that is a labor-and-freshness cost the engineering-free axiom is written to absorb, not a growth of the trusted set. This supersedes the prior "no pure-win" reading, which held only under the unstated assumption of *inheriting* seL4's proof.

> **Revision note (v9).** This version hardens the userspace admission floor from a *trusted* property into a *proven* one, spending "engineering is free" to shrink the trust base (§5, §6, §13, §17, §18). **Tier-2 apps must now carry a binary-level memory-safety certificate** — temporal safety, control-flow integrity, no-runtime-codegen, and ABI/type conformance, stated against the CHERI-RISC-V Sail model and discharged by the same §6 foundational checker that gates the TCB; CHERI already enforces *spatial* safety at runtime, so the certificate proves only the residual. This retires the pre-v9 posture in which app memory safety "leaned on the Rust toolchain": for admitted apps, **rustc/LLVM leave the intra-compartment memory-safety trust base**, matching what FPCC already did for the TCB, and the source-level `#![forbid(unsafe_code)]` attribute is replaced by an artifact-level obligation ("property of the artifact, not its pedigree," §5) that is language-agnostic and cannot be evaded by shipping a non-Rust or hand-crafted binary. The enabling artifact — a **certifying Rust→RV64+CHERI compiler that emits the memory-safety certificate automatically from safe source** — does not exist off the shelf and is **adopted as an in-scope project workstream (§18)**, a sibling prerequisite to the CHERI-CompCert backend (§6); it is a **hard prerequisite for building or admitting any userspace app** — there is no trusted-toolchain fallback, so userspace availability gates on the toolchain exactly as desktop instantiation gates on CHERI silicon (§18). The deliberate tradeoff is recorded in §17: admission now **refuses uncertified code**, narrowing the earlier "hardware bounds arbitrary unverified code" property (§13) in exchange for memory-safety-by-proof; the hardware universal contract (CHERI + caps + IOMMU) is retained *underneath* as defense-in-depth, so containment is unchanged and only the admitted set shrinks. App-developer burden is ~zero for pure-safe-Rust code (the certificate is compiler-emitted); `unsafe`-needing apps must route through the verified HAL (§5) or ship a manual memory-safety proof (Verus/Prusti/Aeneas-class), converting the existing no-app-`unsafe` policy from convention into a hard admission wall.

> **Revision note (v8).** This version adopts a second, tightly bounded role for RISC-V **PMP beyond `Smepmp` self-lockdown: a static, locked-at-boot, *subtractive* M-mode backstop** (§15) that sits *below* the S-mode kernel and is implemented *disjointly* from CHERI — the only mechanism on the die that can deny the kernel itself physical reach (the MMU is kernel-controlled and cannot bound S-mode against itself; CHERI shares the kernel's capability-machine failure modes). It furnishes an independent failure domain for a handful of coarse crown-jewel invariants: the immutable-text/W^X backstop (§14), the per-core physical-partition bound (§7), and crown-jewel secret windows fenced from general S-mode reach. Admissibility is structural — subtractive (denies only, never grants; no new authority path), static and locked (never reprogrammed per switch, so nothing joins the fence.t set and no timing channel opens), and derived from the existing composition artifact (§9), so it is a second enforcement of an existing policy, not a new source of truth. Its independence covers kernel compromise and an independent CHERI/MMU *logic* fault — **not** a shared-Sail specification error or fab compromise (both remain §17 residuals: one Sail model, one mask set). On the purecap platform PMP is admitted *only* in this coarse-static-subtractive role; fine-grained or per-switch-dynamic PMP stays dominated by CHERI (see [Generational Directions](generational-directions.md)). That document delineates PMP's dominated primary role from this non-dominated backstop role, and §18 records PMP's interim coarse-isolation stand-in on the non-CHERI target.

> **Revision note (v7).** This version curates the atomics surface, applying the v6 predictor logic to synchronization. The `A` extension is decomposed: **`Zaamo`** (atomic RMW) is retained, **`Zacas`** (compare-and-swap, incl. 128-bit `amocas.q`) is adopted, and **`Zalrsc`** (LR/SC) is excluded outright (§15). LR/SC's reservation set is hidden microarchitectural state carried *between* two instructions, so it fails admission-test-3 exactly as a dynamic predictor does — and *deleting* it is strictly stronger than flushing it (nothing joins the fence.t set, no residual "was the reservation cleared?" obligation, and the reservation-granule contention signal, a cross-hart channel, vanishes structurally). SC's architecturally-sanctioned spurious failure is the base atomic ISA's one nondeterminism (admission-test-1), and a single fixed-latency AMO beats an LR/SC retry loop for WCET (§11); the reservation set, spurious-failure nondeterminism, and constrained-loop forward-progress rules all leave the Sail model (§17). The removal is a pure win *here* because the trust boundary has almost no LL/SC customer: the share-nothing multikernel holds no kernel locks (§7) and the bounded-SPSC ring data plane synchronizes single-writer head/tail indices without CAS or LR/SC (§12), leaving only contained multi-writer servers, which `Zacas` serves. The sole forfeit — LL/SC's ABA-immunity and CHERI `LR.C/SC.C` atomic tagged-capability update — is covered by `amocas.q` (versioned CAS) and obviated by rings carrying indices, not capabilities ("authority physically cannot cross the data plane," §12). `Zawrs` stays excluded and is now reinforced — it would stall on a reservation set that no longer exists.

> **Revision note (v6).** This version makes two architectural decisions that the non-speculation posture had left implicit, both in §15. (1) **Ztso (RVTSO) is adopted as the normative architectural memory model.** In-order issue plus a FIFO store buffer implements TSO natively at essentially zero microarchitectural cost, and adopting it deletes the RVWMO reordering-reasoning burden from every ring and cross-island proof — the weak model existed only to license the speculative/out-of-order reordering this platform forbids. Full SC was rejected on store-buffer-drain stall cost; Ztso is the fixed point that captures the proof simplification without the stall. All prior "under RVWMO" proof obligations (§12 rings, cross-island rings, the messenger) are restated under Ztso. (2) **Control-flow prediction is now normatively static-only** (backward-taken / forward-not-taken), with **zero mutable predictor state** — no BHT/BTB/RAS or any dynamic direction/target/return predictor. Dynamic prediction is hidden shared microarchitectural state that survives a partition switch (admission-test-3 failure) and a cross-domain history channel; deleting the structures is strictly stronger than flushing them (there is nothing to add to the fence.t set). The IPC tax on forward/indirect branches is accepted and partially recovered by deterministic profile-guided fall-through layout (§10). §17's transient-execution residual is narrowed accordingly, and the §15 anti-features list adds dynamic branch prediction by name.
